# Path C: HMI source vs. TFT bytecode opcode cross-reference

## Goal

Cross-reference the plain-text event-script source committed inside
`source/nextion.hmi.HMI` against the compiled bytecode in
`source/nextion.hmi.tft`, with the aim of identifying previously-undocumented
VM opcodes (the kind that surface in TFTTool output as `op:0x1234`
placeholders).

## Method

1. Read the canonical opcode table from
   [`tools/nxt-doc/Protocols/Full Instruction Set.md`](../tools/nxt-doc/Protocols/Full%20Instruction%20Set.md)
   and the per-(editor-version, model-series) numerated index tables in
   [`tools/TFTTool/NextionInstructionSets.py`](../tools/TFTTool/NextionInstructionSets.py).
2. Run `tools/Nextion2Text/Nextion2Text.py` against the HMI to dump
   per-page event-script source as plain text. (Required two trivial fixes:
   replace the unsupported `decode("ansi")` calls with `decode("latin-1")`.)
3. Run `tools/TFTTool/TFTTool.py` against the TFT to attempt a JSON dump.
   This required adding the editor version `nxt-1.67.1` to the matching
   instruction-set group in `NextionInstructionSets.py` (the latest entry
   in the upstream table is `nxt-1.65.1`).
4. The TFT's header2 is XOR-encrypted with an unknown key for our
   `NX4832F035_011` (Discovery / F series) device, which means TFTTool
   can't read the `usercode_address`/`unknown_pages_address` fields and
   emits an empty `Usercode` section. To work around this, locate the
   usercode region by direct inspection: it begins at `0x70000`
   (immediately after the 0x10000-byte header + 0x60000-byte resources)
   and ends at `0x719a8` (last point at which size-prefixed blocks parse
   cleanly).
5. Re-run TFTTool's `Usercode` parser against `raw[0x70000:0x719a8]` to
   produce a per-block decoded listing.
6. Walk the same byte range with `scripts/opcode_match.py` and
   tally every opcode marker (`09 NN SS`) and system-variable marker
   (`04 VVVVVVVV`) seen, cross-referencing each `(size, num)` pair to
   the matching mnemonic in the instruction-set table.

## Bytecode encoding (recap)

The Nextion VM's instruction stream lives in size-prefixed blocks inside
the TFT. Within each block:

| Byte | Form | Meaning |
|------|------|---------|
| `09 NN SS` | opcode | `NN` = index, `SS` = operand size class (4 or 8) |
| `01 LL LL LL LL` | local var ref | 32-bit local-frame address |
| `05 LL LL LL LL` | global var ref | 32-bit global address |
| `04 VV VV VV VV` | system var ref | low byte = size class, high 24 bits = index |
| `03 LL LL LL LL` | int literal | 32-bit signed |
| `54 20 …` | `jmp` | unconditional jump (op = 0x2054) |
| `2c` | `,` | argument separator |
| `+ - * / = …` | unary/binary operator | ASCII |
| `"` | string literal delimiter | `\` escapes |
| `i` (op 4/0) | conditional jump | a.k.a. `cjmp`; documented hidden opcode |

The `(size_class, index)` -> mnemonic mapping is editor-version and
model-series dependent and lives in
`NextionInstructionSets.all_instruction_sets`.

## Opcode placeholder catalog (this firmware)

**Zero `op:0x...` placeholders are produced for this TFT.** Every opcode
marker resolves cleanly under the model-100 / nxt-1.65.1 instruction
set.

Concretely, the cross-check tallied 112 opcode markers across 13
distinct `(size, num)` pairs in the usercode region, and 15 system-var
markers across 7 distinct pairs. All resolved.

## Worked cross-reference example

HMI source (`work/hmi_text/Program.s.txt`):

```
int sys0=0,sys1=0,sys2=0
baud=115200
recmod=0
printh 00 00 00 ff ff ff 88 ff ff ff
page 0
```

TFT block sequence starting at `0x70004c`:

```
0x07004c (len=11): 04 0e 00 00 00 3d 03 00 c2 01 00
                   |sysvar baud   |=  |literal 115200
0x07005b (len= 7): 04 08 0d 00 00 3d 30
                   |sysvar recmod |=  |'0'
0x070066 (len=32): 09 0b 08 [ascii "00 00 00 ff ff ff 88 ff ff ff"]
                   |opcode printh
0x07008a (len= 4): 09 0b 04 30
                   |opcode page  |'0'
```

Decoded by TFTTool: `baud=0x1c200`, `recmod=0`, `printh "..."`, `page 0`.
Every byte of the bytecode is accounted for. This validates the existing
mapping and answers the cross-reference question for the most direct,
unambiguous block in the firmware.

## Confidence

- **Solid:** every opcode appearing in this TFT maps to a known mnemonic.
  This is verified two independent ways — by running TFTTool's own
  decoder (zero `op:` placeholders in instruction blocks) and by the
  byte-level `opcode_match.py` walk (zero unidentified entries). The
  worked example above shows direct byte-for-byte agreement between
  HMI source and TFT bytecode.
- **Solid:** editor version `nxt-1.67.1` shares the model-100
  instruction set with `nxt-1.65.1`. The decoded usercode is
  self-consistent (operand counts, separators, string contents all match
  the HMI source) when decoded under the 1.65.1 table, so 1.65.1 was
  added to 1.67.1's group in `NextionInstructionSets.py` for this
  worktree.
- **Speculative:** block 0 (`0x070000`, 72 bytes) is a global-memory
  layout descriptor + a tail of `(u32 hash, u16 index)` entries — TFTTool
  mis-decodes it as an instruction block because the bytes happen to
  contain ASCII `<` and a few digits. The decoded-as-instructions output
  for that block is meaningless (`system:0x200000`, `local:0x2d252`,
  etc.) and should not be treated as evidence of unknown opcodes. This
  is a TFTTool heuristic limitation, not a new bytecode finding.

## Still unidentified

- **Opcodes:** zero. This firmware does not exercise any opcode beyond
  the documented set.
- **System variables:** zero. Same.
- **Out-of-scope quirks observed but not investigated here:**
  - Header2 of F-series TFTs is XOR-encrypted with a key TFTTool's table
    lists as `0` for the F series, but our file is clearly encrypted.
    The real key is not yet known — see Path B if that comes up.
  - The "global memory directory" structure of block 0 is undocumented;
    only 4 of the 12 entries look like `(hash, index)` pairs while the
    rest look like layout offsets.

## Implication for nxt-doc

No additions needed for the canonical instruction set as a result of
this firmware. The single instruction-set patch made (adding
`nxt-1.67.1` to the existing 1.61.1/1.63.x/1.65.1 group in
`NextionInstructionSets.py`) is a reasonable upstream contribution if it
holds up across other TFTs compiled with that editor version.

## Reproduce

```
./scripts/setup.sh                                      # populate tools/
python3 tools/Nextion2Text/Nextion2Text.py \
    -i source/nextion.hmi.HMI -o work/hmi_text -n -v    # plain HMI dump
python3 scripts/opcode_match.py                         # opcode catalog
```

(Note: Nextion2Text.py needs `decode("ansi")` -> `decode("latin-1")` and
NextionInstructionSets.py needs `'nxt-1.67.1'` in the
`'nxt-1.61.1', 'nxt-1.63.1', ...` versions list. Both edits live in
gitignored `tools/` and are re-applied each time `setup.sh` is run from
scratch.)
