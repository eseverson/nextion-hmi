# Per-component init bytecode encoder

This document closes [`next-steps.md`](next-steps.md) gap #2 — the
encoder that produces the per-component init bytecode block from a
component (type, x, y, w, h, attrs). The disassembler exists already
([`scripts/tft_bytecode.py`](../scripts/tft_bytecode.py)); this is the
inverse direction.

## Where the emitter lives in the editor

The init bytecode is not produced by a single dedicated routine.
Instead, it is generated by **compiling a per-type Nextion script**
through the regular script compiler. The chain (all in
`/tmp/decoded/plain_hmitype.dll`) is:

1. **`hmitype.mobj::Getatt_Codes(out List<string> mycodes, ev)`** at
   IL offset ~0x16f48 — per-component, per-event-type dispatcher.
   When `xilie.function_objcomref == 1` (F-series / T1), this is the
   active path. Otherwise it falls through to `Getatt_Codes_X`
   which only emits the literal source `"init '&id&'"`.

2. The method holds **one big `switch (objType)` (implemented as
   chained `if`/`ceq` on `appobjs.<field>.id`) → per-event source
   templates**. For each component type, for each event in
   `(init, Ref, editref, down, up, unload, codestimer, ...)`, the
   method appends Nextion script lines (with `'&attrname&'`
   placeholders) to `mycodes`.

3. **`hmitype.mobj::Getbianyi(eventtype)`** at IL ~0x1bc24 — wraps
   `Getatt_Codes`, then for each line calls
   `appbianyi::GetbianyiCodes` which in turn calls
   `appbianyi::bianyionline(obj, errorlabel, &descode, code, ...)`
   to compile that single line into a `byte[]` bytecode block.

4. The output is a `List<byte[]>` — **one bytecode block per source
   line**. The compiler resolves `'&attrname&'` to the attribute's
   address in the per-page attribute-value region; that address
   becomes the LOAD operand in the emitted bytecode.

So the per-component init bytecode at `strdataaddr + PianyiData[+0x34]`
is **not a single length-prefixed block** — it is a **stream of
length-prefixed blocks**, one per compiled source line, followed by
event-label markers (`down`, `up`, `unload`) that separate the next
event's blocks. The "init bytecode" addressed at `+0x34` is really
the start of the Ref-event block sequence.

## Per-type, per-event template table

Below: every observed per-type template, extracted by walking the
IL of `Getatt_Codes`. The dispatch key for most types is `sta` (the
background-fill mode attribute), with `style` (border style) as a
secondary key for types that have it. `'&attr&'` placeholders are
compiled to `LOAD(addr_of(attr))` in the bytecode; the bare integer
literals (e.g. `,0,`, `,1,`) become ASCII bytes in the bytecode. The
4-tuple `(x, y, w, h)` is also inlined as ASCII (since coords are
fixed at compile time).

`Ref` is the event whose compiled block sequence forms the persistent
**init bytecode** (run once when the page loads to draw the
component). `editref` and `down`/`up`/etc. populate other event slots
on the same component.

### `page` (objType 121, label `page`)
```
Ref:     fill '&x&','&y&','&w&','&h&','&bco&'
editref: fill '&x&','&y&','&w&','&h&','&bco&'
```
Plus `pic ...` variant when the page has a background picture.

### `number` (objType 54, label `number`), and `xfloat` (objType 59)
Dispatched on `sta`:
- **sta=0** (transparent background):
  ```
  Ref: setbrush '&x&','&y&','&w&','&h&','&font&','&pco&','&picc&','&xcen&','&ycen&',0,'&isbr&','&spax&','&spay&',0,
  Ref: nstr '&val&','&lenth&','&format&'        # number
  Ref: fstr '&val&','&vvs0&','&vvs1&'           # xfloat
  ```
- **sta=1** (border/style with `style ∈ {1,2,3,4}`, with `style=1`
  pulling `borderw` and falling through to "1" otherwise):
  ```
  Ref: setbrush '&x&','&y&','&w&','&h&','&font&','&pco&','&bco&','&xcen&','&ycen&',1,'&isbr&','&spax&','&spay&',0,<borderw>
  Ref: nstr|fstr ...
  ```
- **sta=2** (background picture):
  ```
  Ref: setbrush '&x&','&y&','&w&','&h&','&font&','&pco&','&pic&','&xcen&','&ycen&',2,'&isbr&','&spax&','&spay&',0,
  Ref: nstr|fstr ...
  ```
- **sta=3** (text only, no fill):
  ```
  Ref: setbrush '&x&','&y&','&w&','&h&','&font&','&pco&',0,'&xcen&','&ycen&',3,'&isbr&','&spax&','&spay&',0,
  Ref: nstr|fstr ...
  ```

### `text` (objType 116) and `gtext`/ScrollingText (objType 55)
Same `sta` dispatch as `number/xfloat`, but:
- The setbrush trailing argument is `'&pw&'` for `text` and `0` for `gtext`
- Final draw is `zstr '&vvs0&','&vvs1&','&vvs2&','&vvs3&','&txt&'` for
  `gtext` and `zstr 32767,32767,32767,32767,'&txt&'` for `text`

### `button_t` (objType 53, DualStateButton) and `button` (objType 98)
Dispatched on `sta` (and `style`). Bodies wrapped in
`if('&val&'==1) { ... } else { ... }` to swap pressed/released colors:
```
Ref: if('&val&'==1)
Ref: {
Ref: setbrush ...,'&pco2&','&picc2&',...,0,'&isbr&','&spax&','&spay&',0,
Ref: }else
Ref: {
Ref: setbrush ...,'&pco&','&picc&',...,0,'&isbr&','&spax&','&spay&',0,
Ref: }
Ref: zstr 32767,32767,32767,32767,'&txt&'
```
Plus `style=4` adds a `draw3d ...` border.
For `button` the down/up events also include `'&val&'=N` and
`ref '&id&'+65535`.

### `checkbox` (objType 56) and `radiobutton` (objType 57)
- Checkbox:
  ```
  Ref: sya0='&w&'/4
  Ref: sya1='&w&'-sya0-sya0
  Ref: fill '&x&','&y&','&w&','&h&','&bco&'
  Ref: if('&val&'==1)
  Ref: {
  Ref: fill '&x&'+sya0,'&y&'+sya0,sya1,sya1,'&pco&'
  Ref: }
  ```
- Radio:
  ```
  Ref: sya0='&w&'&1
  Ref: if(sya0==0) { sya0='&w&'/2-1 } else { sya0='&w&'/2 }
  Ref: cirs '&x&'+sya0,'&y&'+sya0,sya0,'&bco&'
  Ref: cir '&x&'+sya0,'&y&'+sya0,sya0,16936
  Ref: if('&val&'==1) { cirs '&x&'+sya0,'&y&'+sya0,sya0/2,'&pco&' }
  ```

### `qrcode` (objType 58)
```
Ref: qrcode '&x&','&y&','&w&','&bco&','&pco&',<pic_or_65535>,'&dis&','&txt&'
```
The 6th argument is `65535` when `pic == 0`, else `'&pic&'` (literal
picture index).

### `pic` (objType 112)
```
Ref: pic '&x&','&y&','&pic&'
```

### `picc` (objType 113, q0 / cropped picture)
```
Ref: xpic '&x&','&y&','&w&','&h&','&x&'+'&vvs0&','&y&'+'&vvs1&','&picc&'
```

### `prog` (ProgressBar, objType 106)
Dispatched on `dir` (and `sta`):
- Horizontal fill, no picture:
  ```
  Ref: sya0='&val&'*'&w&'/100
  Ref: fill '&x&','&y&',sya0,'&h&','&pco&'
  Ref: fill '&x&'+sya0,'&y&','&w&'-sya0,'&h&','&bco&'
  ```
- Vertical, picture-fill, etc. — see [`Getatt_Codes` source]; multiple
  branches with `xpic ...,'&ppic&'/'&bpic&'`

### Empty / no-Ref types
These emit no Ref-event blocks (or only event-handler blocks):
`hotspot/touch (109)`, `timer (51)` (emits a `timerset` line in its
own `timer` event), `variable (52)`, `slider (1)`, `waveform (0)`,
`croppicture (5)`.

For these types the disassembler shows the init bytecode block as
empty (length 0) — confirmed in [`format-bytecode.md`](format-bytecode.md).

### Other observed lines (event handlers, not Ref)
- `editref` event: same shape as `Ref` but with `,3,` mode (edit
  preview) and an extra `xpic`/`pic`/`fill` overlay.
- `down`/`up` events: `'&val&'=N` toggles for buttons/checkbox/radio,
  plus `ref '&id&'+65535` to trigger redraw.
- `codestimer` event for timers: `S 1'&pageid&'12'&id&'1` (a
  hardware-specific timer-arm sequence).

## Resulting bytecode shape

After compiling each source line via `bianyionline`, each line becomes
one length-prefixed bytecode block of the form:

```
<u32 length>
<3-byte opcode header>          # editor-specific; observed values:
                                 #   09 1d 08  setbrush (size 8)
                                 #   09 14 04  fstr/nstr (size 4)
                                 #   09 12 04  fstr (variant)
                                 #   09 0d 04  fill (size 4)
                                 #   09 0a 04  xpic (size 4)
                                 #   09 01 04  pic (size 4)
                                 #   09 08 08  qrcode (size 8)
                                 #   04 04 05  checkbox-shorthand
<args body>                      # comma-separated; literals as ASCII,
                                 # attrs as 5-byte LOAD ops
```

Within args, the body is built by walking the template left-to-right
and emitting either:
- a 5-byte LOAD operand for each `'&attr&'`: `01 LL HH 00 00`,
  where `0xHHLL` is the **u16 record index** of the attribute's value
  in the per-page 24-byte-stride attribute-record table at
  ``strdata + pagexinxi.attdataaddr`` (resolved via
  [`scripts/tft_attrs.py`](../scripts/tft_attrs.py) — agent 1's
  territory, see [`attribute-records.md`](attribute-records.md)),
- a 5-byte long-form int literal `03 LL HH HH HH` for compile-time
  numeric constants that don't fit a short ASCII form (e.g. the
  `65535` placeholder in qrcode's "no picture" branch),
- an ASCII byte for every other character in the template (digits,
  literal `0`/`1`, operators `+`, `-`, etc.),
- a `2c` (`,`) for each comma.

`'&x&'`, `'&y&'`, `'&w&'`, `'&h&'`, `'&id&'`, `'&pageid&'`,
`'&lenth&'`, `'&format&'`, `'&dis&'`, `'&en&'`, `'&tim&'` are inlined
as ASCII decimal literals at compile time (they are constants from the
component definition, not dynamic attributes). Everything else
(`&font&`, `&pco&`, `&bco&`, `&val&`, `&vvs0..3&`, `&picc&`, `&pic&`,
`&xcen&`, `&ycen&`, `&isbr&`, `&spax&`, `&spay&`, `&pw&`, `&style&`,
`&borderw&`, `&txt&`, `&ppic&`, `&bpic&`) emits LOAD operands.

## Verification against `17_more_components/17.tft`

For XFloat `x0` at `(0, 20, 160, 50)`, sta=1, style=4:
- Block at file offset `0x800e2` (length 68):
  ```
  09 1d 08 0 , 20 , 160 , 50 ,
           LOAD(55) , LOAD(57) , LOAD(56) , LOAD(58) , LOAD(59) ,
           1 ,
           LOAD(63) , LOAD(64) , LOAD(65) ,
           0 , 0
  ```
  Matches `setbrush x,y,w,h,font,pco,bco,xcen,ycen,1,isbr,spax,spay,0,0`.
- Block at `0x8012a` (length 20):
  ```
  09 14 04 LOAD(60) , LOAD(61) , LOAD(62)
  ```
  Matches `fstr val,vvs0,vvs1`.
- Block at `0x80146` (length 20):
  ```
  09 0d 04 0 , 20 , 160 , 50 , LOAD(24)
  ```
  Matches `fill x,y,w,h,bco_page` (page background bco at attr
  offset 24 — shared across all components on a page).

For the next XFloat `x1` at `(160, 20, 160, 50)`:
- LOAD addresses are shifted by +41 (per-component stride in the
  attribute-value region): LOAD(96), LOAD(98), LOAD(97), LOAD(99),
  LOAD(100), `1`, LOAD(104), LOAD(105), LOAD(106), `0`, `0`; and
  LOAD(101), LOAD(102), LOAD(103) for the fstr.

The LOAD operand is a **u16 record index** into the per-page
24-byte-stride attribute-record table at
``strdata + pagexinxi.attdataaddr`` — see
[`scripts/tft_attrs.py`](../scripts/tft_attrs.py) and
[`findings/attribute-records.md`](attribute-records.md) (agent 1's
output) for the table layout. The per-component "stride" of ~41
records reflects how many distinct attribute slots each XFloat
allocates; the actual byte layout of those records is fully decoded
in `tft_attrs.py`.

## Encoder algorithm

```python
def encode_init_block(comp_type: int, x, y, w, h, attrs: dict,
                      attr_addr: Callable[[str], int]) -> bytes:
    """Emit the full init-bytecode stream for one component.

    Returns the concatenation of length-prefixed blocks for the Ref
    event (which is what's addressed by PianyiData[+0x34]).

    `attr_addr` resolves an attribute name → its LOAD address in the
    per-page attribute-value region. This is gap #1's responsibility.
    """
    template = TEMPLATES_BY_TYPE_AND_STA(comp_type, attrs.get('sta', '0'))
    out = bytearray()
    for source_line in template:
        rendered = render_template(source_line, x, y, w, h, attrs)
        bytecode_body = compile_line_to_bytes(rendered, attr_addr)
        out += struct.pack('<I', len(bytecode_body)) + bytecode_body
    return bytes(out)
```

The two non-trivial helpers:

### `render_template(line, x, y, w, h, attrs)`
Just substitutes constant placeholders (`&x&`, `&y&`, `&w&`, `&h&`,
`&id&`, `&pageid&`, `&format&`, `&lenth&`, `&dis&`, `&en&`,
`&tim&`) with decimal-string values. Leaves dynamic-attr placeholders
(`'&font&'`, etc.) intact for the next pass.

### `compile_line_to_bytes(line, attr_addr)`
Recognises the leading mnemonic (`setbrush`, `nstr`, `fstr`, `zstr`,
`fill`, `xpic`, `pic`, `qrcode`, `cir`, `cirs`, `draw3d`, `if`,
`{`, `}else`, `}`, sysvar assignments like `sya0=...`, `ref ...`,
`timerset ...`), maps it to its 3-byte opcode header, and emits the
arg body. Arguments are walked left-to-right:
- A literal numeric token → ASCII bytes.
- A `&attr&` token → 5-byte `01 LL HH 00 00` LOAD.
- A `,` → `2c`.
- An operator (`+`, `-`, `*`, `/`, `&`, `==`, ...) → ASCII.

For control-flow lines (`if`, `{`, `}else`, `}`), the compiler emits
the corresponding jump opcode (`09 00 04` cjmp / `54 20 03` jmp) with
patched branch targets after the block is closed. See
[`format-bytecode.md`](format-bytecode.md#control-flow).

## Opcode-mnemonic mapping in 1.67.1 model 100

The instruction-set table in TFTTool covers nxt-1.65.1 but observed
opcode bytes in our 1.67.1 fixtures differ slightly. Empirical
opcode → mnemonic for the init-bytecode bodies we see in
`17_more_components/17.tft`:

| Bytes        | Mnemonic                                       |
|--------------|------------------------------------------------|
| `09 1d 08`   | `setbrush` (14-arg set-render-state call)      |
| `09 14 04`   | `fstr` (3-arg formatted-string draw)           |
| `09 12 04`   | `fstr` variant (alternate firmware)            |
| `09 13 04`   | `nstr` (3-arg numeric-string draw)             |
| `09 0d 04`   | `fill` (x,y,w,h,colour)                        |
| `09 0a 04`   | `xpic` (cropped-picture draw)                  |
| `09 01 04`   | `pic` (full-picture draw)                      |
| `09 08 08`   | `qrcode` (8-arg)                               |
| `09 15 04`   | `zstr` (5-arg z-axis-bounded text draw)        |
| `09 17 04`   | `addt` (XFloat / Gauge helper)                 |
| `09 04 04`   | `cir` (4-arg circle outline)                   |
| `09 16 04`   | `cirs` (4-arg circle solid)                    |
| `09 07 08`   | `draw3d` (4-arg + colour pair, 3D bevel)       |
| `04 04 05`   | Checkbox compact form (`=N*W/100`)             |

The `09 1c 08` / `09 1d 08` mismatch with nxt-1.65.1's table
(`1c=setbrush, 1d=ref_stop`) reflects a one-slot shift in 1.67.1's
size-8 opcode list. The encoder uses the observed byte values from
1.67.1 fixtures; for older firmware the byte should be `09 1c 08`.

## What this enables and what it still doesn't

**Enables**: given a complete attribute-address map (gap #1's
output), we can emit byte-for-byte-correct init bytecode for every
component type observed in real projects. The encoder in
[`scripts/tft_init_encoder.py`](../scripts/tft_init_encoder.py)
takes that map as a parameter (via the `attr_addr` callable) and
needs no other inputs.

**Still missing**:

1. **Attribute-address resolution** (gap #1): the LOAD operand
   values depend on where each attribute's value lives in the
   per-page attribute-value region. The encoder fills in the
   shape; the actual offsets need the value-table writer.

2. **Per-firmware opcode byte assignment**: the table above is for
   1.67.1. Older firmwares shift `setbrush`/`ref_stop` by one slot.
   For multi-firmware authoring, parameterise the encoder by
   firmware version.

3. **`Getatt_Codes` minor dispatch branches**:
   - The `borderw`-conditional inside sta=1: when `style ∈ {1,2,3}`
     the trailing arg is `'&borderw&'`, when `style ∈ {4}` it's
     literal `"1"`. Implemented in the encoder.
   - The full `if/else` rewriter for `button`/`button_t`/`checkbox`/
     `radiobutton`: control-flow compilation depends on the
     script-bytecode compiler (gap #3, agent 3's territory). The
     encoder emits the source string; final byte output for these
     types needs the script compiler.

4. **Event handlers beyond Ref**: `editref`, `down`, `up`, `unload`
   are also part of the per-component data stream after the Ref
   blocks. They follow the same compile-via-bianyionline pipeline
   but reference event-label markers (`"down"`, `"up"`, `"unload"`)
   stored as ASCII strings between block sequences. Implementing
   these is a strict extension of the Ref encoder.
